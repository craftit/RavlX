The files in the Example subdirectory can be compiled and run to demonstrate a simple example of a two module system. The UnWarp module performs geometric distortion correction to a sequence of video frames, and the ShotDet module does shot detection. The program name is Ingest, and can be run from the source directory without arguments. (See Ingest -help for options.) For example: 

  Ingest

will specify that the shotBoundary module is to be run. This module requires input from the frame module, which will therefore be started automatically.

  Ingest -T frame

will specify that the frame module is to be run. This does not require any other modules to be run. If we then run:

  Ingest -mi /tmp/vampiremem

this will first load up any module results previously stored in the default directory /tmp/vampiremem (in this case the frame) results from the last run) and then specify that the shotBoundary module is to be run. Although shotBoundary requires results from frame, these results have been already loaded in from the previous run, so in this instance the frame module does not need to be run. Because the -mo was not used, the new results will also be stored in /tmp/vampiremem.

This is a very basic example. In practice, a module can read data synchronously from several other modules. Where data items from different source modules have widely differing time frames, the fine-scale data loop can be nested within the coarse-scale data loop. See here for a more complex example.

The testMemModules.cc test program provides an example of how several modules can communicate with each other. This example includes an illustration of how feedback might be implemented; however avoiding deadlock is the responsibility of the programmer.
